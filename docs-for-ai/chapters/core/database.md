# Database — SQLite connection, schema, and migrations

## Overview

The database layer provides a singleton SQLite connection through Drizzle ORM, defines all table schemas, and handles migration execution. SQLite runs in WAL mode for concurrent read access. All timestamps are stored as integer milliseconds since epoch.

## Key Behaviors

- **Lazy Initialization**: `getDb()` creates the connection on first call, reused thereafter. Module-level `db` variable holds the singleton.
- **WAL Mode**: `pragma("journal_mode = WAL")` enables concurrent readers with serialized writers.
- **Foreign Keys**: `pragma("foreign_keys = ON")` enforces referential integrity at the SQLite level.
- **Config**: `DATABASE_PATH` env var controls DB file location (default `./data/rss-agg.db`).

## Interface

`getDb()` — returns `AppDatabase` (Drizzle instance with schema).

`runMigrations()` — applies pending migrations from `packages/core/drizzle/migrations/`.

`type AppDatabase` — `ReturnType<typeof getDb>`, the typed Drizzle client.

## Internal Details

- **Migration path resolution**: Uses `import.meta.url` to resolve `../../drizzle/migrations` relative to the compiled JS output, not the TS source.
- **ID generation**: All PKs use `nanoid()` (21-char URL-safe random string), generated by each service at insert time.
- **Timestamps**: All `created_at` columns use SQLite `unixepoch() * 1000` default; application-set timestamps (e.g. `lastFetchedAt`, `readAt`) use `Date.now()`.

## Dependencies

- Uses: `better-sqlite3`, `drizzle-orm`, `drizzle-orm/better-sqlite3`, Logger
- Used by: Feed Service, Entry Service, Category Service, AI Service, Scheduler Service (indirectly via Feed Service), API routes (direct queries), Migration Runner (API init, CLI init)

## Threading / Concurrency

SQLite WAL mode: multiple concurrent readers, single serialized writer. No application-level locking. Write contention handled by SQLite's busy timeout (default).
